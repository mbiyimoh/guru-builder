**Slug:** wizard-dashboard-integration
**Author:** Claude Code

**Branch:** feat/wizard-dashboard-integration

## 1) Intent & Assumptions

**Task brief:** Integrate the wizard flow's best features (readiness scoring, gap analysis, research suggestions, artifact generation) into the new dashboard-anchored experience, eliminating the narrow-viewport linear wizard while preserving its guided content.

**Assumptions:**
- The SimplifiedDashboard at `/projects/[id]` is the new "home base" for all project work
- Users want access to all features without being forced through a linear sequence
- The valuable UX content (readiness scores, gap suggestions, chat assistants) should be preserved
- Full-width layouts are preferred over the constrained `max-w-4xl` wizard pattern
- Users may jump between sections in any order (non-linear)

**Out of scope:**
- Publishing functionality (marked as "Coming Soon")
- Mobile-specific redesigns beyond responsive patterns
- Changes to core API endpoints
- Changes to underlying data models

## 4) UX Analysis: Current vs. Proposed

### Current Wizard Experience - What's Great

| Feature | Location | Value Proposition |
|---------|----------|-------------------|
| Gap-aware research suggestions | `/new/research` | Shows critical/suggested gaps with "Research This" buttons |
| Chat assistant for research | `/new/research` | Split-panel chat + plan display |
| Active research polling | `/new/research` | Shows research-in-progress with status |
| Readiness scoring | `/new/readiness` | Overall score, profile/knowledge breakdown |

| Critical gap CTA | `/new/readiness` | "Research This" buttons inline |
| Artifact generation | `/new/artifacts` | 3 artifact cards with dependency chain |
| Guru test chat | `/new/artifacts` | 20-message ephemeral testing |

### Current Wizard Experience - What's Problematic

| Issue | Location | Impact |
|-------|----------|--------|
| Narrow viewport (`max-w-4xl`) | `new/layout.tsx:8` | Wastes screen real estate |
| Linear forcing | WizardNavigation | Can't access Readiness without going through Research |
| Disconnected from project | All `/new/*` | Uses `?projectId=X` instead of `/projects/[id]/*` |
| Duplicate navigation | WizardNavigation | Separate nav from main app header |
| No breadcrumb to dashboard | All wizard pages | Hard to return to project home |

## 8) Research Findings

### Approach Comparison

| Approach | Pros | Cons |
|----------|------|------|
| **A: Full Migration** - Delete wizard routes, use only `/[id]/*` | Clean architecture, single source of truth | Breaking change, need to handle truly new projects |
| **B: Parallel Routes** - Keep both wizard and dashboard routes | Gradual migration, backward compatible | Duplicate code, maintenance burden |
| **C: Redirect Strategy** - Wizard routes redirect to dashboard routes | User-friendly transition, no broken links | Extra redirect layer, slight complexity |

### Recommendation: Approach C (Redirect Strategy)

1. Create new `/projects/[id]/{research,readiness,profile}` routes
2. Update dashboard links to use new routes
3. Add redirects: `/projects/new/*?projectId=X` â†’ `/projects/X/*`
4. Keep `/projects/new/profile` for initial project creation (no ID yet)
